Software Analysis

Use Cases

Use Case 1:

Use Case Name:

New Student User

Participating Actors:

New Student User
Student Community (Other student users)

Flow of Events:

  1. New student tries to log in, if login fails or the account doesn’t exist, the user is prompted to sign up.
  2. If the login succeeds, credentials are verified, and account information is retrieved from the database. If login fails, a login error is displayed.
  3. User can then add classes, events, and/or assignments to their TasQ application.
  4. User can always select courses, events, and/or assignments from the crowdsourced information which is retrieved from the database.
  5. If the user wants to add new information, they can do so which will add the new data to the database and update it with new information
  6. This update will also be reflected in the crowdsourced information database and will be communicated with the rest of the student community.

Entry Condition:

User either has a valid account with TasQ or is willing to sign-up with TasQ.

Exit Condition:

The student has created a valid TasQ account with appropriate information.

Quality Requirements:

  1. The student is able to create an account within 30 seconds.
  2. The student is able to create classes, events, and/or assignments within 30 seconds.
  3. Changes made to classes are updated in the database and communicated accordingly with every student in the student community.
  4. TasQ is able to load previous account details of a user within 10 seconds.


Use Case 2:

Use case name: Existing User Creating a New Course

Participating Actors:

Initiated by Student

Flow of Events:

  1. Student chooses “Add course” from the Profile page of the TasQ application.
  2. TasQ responds by offering a list of existing coursees to the Student, as well as a space to create their own course.
  3. The student chooses to create a new course, and provides the course code, name, term and year for the course, and the professor of the course.
  4. The new course is added to the TasQ database.
  5. The student clicks “Add Event” to schedule events for the course, then provides a name, day, time, location, and type of event.
  6. The course in the database is updated with the newly created event.
  7. The student creates assignments for the course on the Assignments page of the TasQ assignment by selecting the plus button, attaching it to the course, and providing a name, due date, and type of assignment.
  8. The course in the database is updated with the newly created assignment.

Entry Conditions:

The student is logged into their TasQ account.

Exit Conditions:

The student has created their own course, with associated events and assignments.

Quality Requirements:

  1. The student is able to create the course, an event, or an assignment in less than 1 minute.
  2. The database is updated within 30 seconds of implementing the change.


Object Models

Encapsulation

The purpose of the API in this software is achieved through information hiding or data hiding to be used to reduce complexity and increase reusability. The user cannot see the inside of an object or a class, but the object can be accessed by calling the object’s methods. At the same time, the background database can protect users' privacy from being seen by other users to the greatest extent.The software also will have network functionality through user profiles and crowdsourced data. By allowing users to create profiles, their courses, assignments, and events can be synchronized across computers or easily recovered if their computer is lost. 

Hierarchy

A hierarchy class is composed of a base class (parent class) and derived classes (subclass). A derived class inherits the properties of a parent class. In the software base class is calendar view & reminders, derived classes are assignments/events, filtering & sorting, to-do list.

Modularity

The software divides the Base class and the Derived class into components, all made into GUI. Modularity focuses on implementation and is convenient for users to use. This software has a lot of classes, modular will be easy to manage and maintain. Each module is switched without delay when used by the user.

System Design

Design Goals

Design goals in software development are system qualities that need to be optimized and are built on the non-functional requirements discussed in the RAD. Design goals focus on five major categories of software: Performance, Dependability, Cost, Maintenance, End User Criteria. 

Performance

The performance of a software is an essential quality of every software system and must be optimized to make applications work efficiently and execute rapidly. Performance relates to a software’s response time, throughput and memory. 

Response time is the amount of time between an inquiry on a system and the response or fulfillment to that inquiry [1]. In today’s day and age, society desires short response times from modern systems. As response time increases, user satisfaction decreases, however, instant response does not signify a user-friendly and easy-to-use system [2]. According to dns, a one-second response time is the maximum acceptable limit as users will not likely notice the delay [3]. TasQ will use an optimized algorithm with high efficiency to ensure a short response time.

Throughput refers to the number of tasks a system can complete in a fixed time slot and can be calculated using the following equation: Throughput = total successful units produced / total time. High throughput is generally ideal as it indicates a more efficient system. There are certain factors that affect a system's throughput, one being network congestion. The more users TasQ will have, the higher the network congestion, the lower the throughput. TasQ will monitor network traffic to look out for bottleneck patterns to ensure high throughput. 

Memory is used to store information for immediate use in a system. Systems that occupy large amounts of memory take up a lot of CPU resources, which affect the efficiency and performance of operations. The memory usage for TasQ should be controlled below 50% as if it is above this threshold it may freeze users' laptops. To ensure system efficiency, user accounts will be closed and treated as garbage data if they are inactive for more than two years to reduce the system workload.  

Dependability

The dependability of a system portrays the user’s degree of trust in that system and confidence that it will operate as expected. The dependability of a system reflects a system’s robustness, reliability, availability, fault tolerance, security, and safety. 

The robustness of a software is the degree to which a system can function in the presence of invalid inputs [4]. TasQ will handle this by throwing exception handlers where necessary and displaying all relevant error messages to users to change invalid inputs. 

Good reliability of a software describes that the observed behaviour matches the specified behaviour. It is a customer-first view of software quality and ensures the operation of a software in comparison to design [5]. The team will maintain a user-first approach when developing TasQ to ensure the reliability of the system. 

Availability of a software relates to the percentage of time the system is available to perform normal tasks and can be calculated using the following formula: Availability = Uptime / (Uptime + Downtime). Good availability is considered to be 90% or higher [6]. 

Fault tolerance relates to a software's ability to continue operations under erroneous conditions. To ensure TasQ can continue normal operations when components fail, the system will have backup components to automatically take the place of failed components. 

The security and safety of a software describes its ability to withstand malicious attacks and its avoidance of human injury in presence of failures. TasQ will require a backup of the database to ensure users could recover all personal information. Users are less likely to use low-level security software, so it is obligatory for TasQ developers to ensure user privacy.

Cost

Cost is the main driving factor for all systems as it largely affects a project’s success rate. The percentage of projects that exceed the estimated cost is 69% whereas the percentage of projects having accurate cost estimation is 31% [7]. Software cost relates to developmental cost, deployment cost, maintenance cost, and administration cost. 

The development cost of a system refers to the cost of initial system development and is directly proportional to the size of the software and its user base. Deployment costs refer to the cost of installation and user training. Maintenance cost relates to the cost of bug fixes and enhancements. Administration costs refer to the cost to administer a system. The following information is taken from FRESHCODE  [8].

The estimation process of development is lengthy and requires lots of data analytics.  Firstly, it is required to estimate the time the project will take to complete. A basic software application, like TasQ, will require a developmental process of 1-2 months. Secondly, it is essential to address the 5 key factors which affect software cost estimations: software size, complexity, UI/UX design, team, platform. Software and user sizes are defined by the number of pages it has and the number of end-users it is expected to adopt. The number of features of an app defines its complexity. TasQ will have a range of 5-10 features. The UI/UX cost of a software ranges as it depends on the app type. Maintenance

Software maintenance is the process of modifying and adapting a system to ensure successful user stories and is a result of changes in demand. After a software product is released it will likely require modifications for correcting errors, improving performance or any other attributes. Software maintenance focuses on a software’s extensibility, modifiability, adaptability, portability, readability, and traceability of requirements. 

The extensibility of a software refers to how easy it is to add functionality or new classes to the system. Modifiability of a software refers to how easy it is to change functionality. The better the extensibility and modifiability of a software, the higher the user satisfaction, as TasQ will be able to alter designs to fit user needs. 

Portability of a software system refers to its ability to port the system to different platforms. A software can increase its popularity once it can be operated on different platforms. Due to TasQ’s target audience being university students, it will be readily available on both Windows and Macs. 

A software’s readability refers to how easy it is to understand the system from reading the code. Traceability of requirements refers to the ability to map the code to specific project requirements. These are both vital pieces of a software’s maintenance as it ensures all other listed maintenance requirements can proceed. All software developers will be required to add comments and explanations where necessary, either in the actual code or in separate documentation to ensure its readability and traceability.

End-User Criteria

End-user criteria refer to the tasks that end-users need to be able to carry out and are split into utility and usability. The utility of a software relates to how well the system supports the work of the user. Whereas, usability relates to how easy it is for the user to use the system. These are both vital components of software development as they ensure successful user stories. TasQ will be designed in a very user-friendly way to guarantee the effectiveness, efficiency, and overall satisfaction of users.

Subsystem Architecture

Subsystem Decomposition

The system is divided into four core subsystems: the Central Server, Network Interface, State Management, and User Interface. The below figure shows a black-box view of the major subsystems and their communication. The following figure shows a grey-box view of the subsystems to better demonstrate the main internal and external communication.

User Interface

The user interface subsystem consists of the Qt window and it's widgets. This is the system through which users interact with the system, and consists of all major boundary objects. The subsystem consists of just the QWidgets used in building the interface to improve cohesion. The interface communicates with the state management system by sending and receiving Qt signals and slots. This ensures minimal coupling, as changes to the interface or state management system's operation only require slight changes to how signals are triggered.

State Management

The state management subsystem contains the classes which track and control the application state while it's running. It also includes components for serializing and deserializing locally stored data. The subsystem communicates with the frontend using bidirectional signal/slot connections. It also communicates with the network interface by calling functions to send or receive data from the central server as needed, primarily through the network models. While the storage and state management classes could be split into separate subsystems for improved cohesion, their high level of coupling favors the combined subsystem.

Network Interface

The network interface is the network-facing subsystem for the application. It responds to function calls from the state management system, and issues HTTP requests to the central server using the provided data. The JSON responses are then parsed and the results are returned to the calling code. These communication methods allow for minimal coupling, as changes in the API are insulated from the applications state management and vice-versa.

Central Server

The central server is a LAMP server which is used to store data for user profiles and crowdsourcing. It consists of the central database and a PHP server which provides an API. It communicates with the application through HTTP requests to the API, which return responses in a JSON format.

Persistent data management 

The system will store various persistent data in two locations. The primary storage will be on a central server, which will allow for user accounts and crowdsourcing. Data pertinent to the local user will also be serialized and stored in local files to allow for offline users. The synchronization process between central and local storage is described in Global Software Control.

Models

The data stored by the system is divided into 9 models:

  • User: Represents a user account/profile
  • Session: Represents a user's session in the system (central db only)
  • School: Represents a school that is registered in the system
  • Course: Represents a course, attached to a school and owned by a user
  • Event: Represents an event attached to a course
  • Subscription: Forms a link between a user and a course they're subscribed to (central db only)
  • Reminder: Represents a reminder for an event set by a user
  • Todo: Represents a To-Do item created by a user
  • Application: Stores application state data (local storage only)

Comprehensive model information including full data schema and SQL table creation statements can be found in Appendix A.

Central Database

The central database is a MySQL database running on a remote server. It can be accessed through a REST API written in PHP. This database stores all data that may be accessed by the application, including users, courses, events, etc. The centralization of this data on a remote server allows for the user profile and subscription features offered by the application, as devices can obtain the information they need from it. MySQL was chosen for it's high reliability, strong structure, and team member familiarity.

Local Storage

The local storage is a series of files on the user's device. These files contain serialized versions of a subset of the centralized database which is used by the local user, so it can be accessed without an internet connection. While the data stored is largely similar to the central database, some information is omitted for privacy and efficiency. These include users' email addresses and passwords for security, and the subscription model as any courses stored locally are assumed to be subscribed. The local storage will also contain additional application data, which contains state data such as session and synchronization information.

Access control and security

User authentication is performed using a session token method. When a user logs in, they are issued a unique token to be stored by the device. This token is then passed in a cookie to future requests, and those which need authentication use it to confirm the identity of the user. The token expires after a set time, requiring the user to log back in. The session token method was selected as it provides an easy way to sessions without storing private credentials, and as the token is only used for authentication it can be discarded at any time to stop malicious access. Access control is performed based on an owner/consumer model.  When a user creates data in the central database, they become its owner. Any other users accessing that data are considered to be consumers. The table below shows which permissions are afforded to owners and consumers of the various data models. Standard database security procedures are followed, with passwords being stored as salted hashes to protect against leaks and TLS being used on all API requests to protect against man-in-the-middle attacks. All these security measures are performed by the API, so while unauthorized changes can be made on the local storage database, this will only impact the stability of that user's application with no impact on the central database.

Global software control

Global software control is achieved through the data storage logic in the application. As the API is RESTful, all communication must be initiated by the application. As application data is stored both remotely and locally, this can lead to synchronization issues when the central database is updated by another device. Synchronization with the database will be achieved through flags and timestamps. When a user modifies data in the application, a flag in the local storage is set indicating there is data to be uploaded. When the application is next able to communicate with the API, it will detect this flag and upload all modified entities for which the user has adequate permissions. If the data flag is not set when connection is established, the application will perform a pull action, getting all reminders and todos. As courses represent the bulk of user data and are unlikely to see frequent changes, the application will request that the API only send data on those which have been modified since the last pull was performed, using a timestamp saved locally. The application will perform periodic data checks to ensure the local data is the most up to date and reduce the likelihood conflicts occur. In the event of a conflict where both the central database and local storage have changes, the local storage will take precedent and updates will be pushed regardless of what is on the server. While this conflict resolution strategy may occasionally result in the loss of data, it will not occur unless the users make changes on separate devices simultaneously without internet connection.

Boundary conditions

Startup – No state data

When the application starts for the first time, or following unexpected data loss due to external events, there will be no state data to restore. In this event, data must be fully initialized from network sources. For this reason, if no API connection can be established the application will not proceed. Once a connection has been established, the user will be prompted with a login/registration form. Input data will be passed to the AuthController for validation. Once a session has been obtained, the ApplicationContoller will request data from the StorageController, which will use the APIController to begin pulling data from the server to initialize the models. For existing users, this will pull their subscribed courses, todos and reminders. For a new user, they will be prompted to subscribe to courses from their selected school or create a new one.

Startup – State data present

Upon startup, the application must restore it's state from stored data. The StorageController will read in stored data from disk using the Deserializer boundary class. This data will then be sent to the ApplicationController to initialize the user interface. Once the state has been loaded, the APIController will begin attempting to establish a connection.

API Connection Established

When the connection is established with the API, the application will synchronize with the remote database per the procedure outlined in Global Software Control. The APIController (at the request of the ApplicationContoller) will attempt to establish a connection immediately following the startup procedure, however, this may occur at any point after a loss of connection. In this case, the application will discreetly indicate to the user that synchronization is occurring.

API Connection Lost

When the connection with the API is lost, the application will discreetly indicate to the user that synchronization cannot occur, and the StorageController will only store changes locally until connection is reestablished.

API Refuses session

If for a variety of reasons the API rejects a user's token when a request is made, the application will display the login screen, prompting them to log back in and create a new session.

API Error occurs

If an API error occurs, it will be displayed to the user as an error message. Any future action must be decided by the user based on error contents and available documentation.

Shutdown

When the application is shut down, the StorageController will save state information to be restored on the next start. If there is local data that has not been uploaded, the APIController will make a final connection check and upload attempt before shutting down.

Unexpected shutdown

As most state data will be saved immediately after changing, unexpected shutdowns outside of the saving procedure should not have any impact on data. Shutdowns during these procedures will have undefined behaviour as the application may fail during a storage update.

Subsystem Services

User Interface

The user interface subsystem provides the end user with an interface into the application. It enables CRUD operations as well as sorting and filtering for profile, schedule, assignments, and todos through a simple graphical interface.

State Management

The state management subsystem acts as the single source of truth for all data that hydrates the GUI. It offers signal/slot connections to the views, enabling subscription-based redrawing of components when data in its view is modified. It also handles saving state to local storage and loading state from the local storage on startup, to enable offline use. And it also handles synchronization with the central server.

Network Interface

The network interface subsystem exposes functions that are used by the state management subsystem for synchronization with the central server. Exposed functions issue HTTP requests to the central server to perform CRUD operations on a user's data, and return to the caller the JSON data that is returned by the server. 

Central Server

The central server subsystem exposes REST endpoints that are used by the frontend network interface. This REST server enables synchronization and persistance of application data through the Apache webserver and MariaDB SQL database.

CREATE TRIGGER `course_mod_ins` AFTER INSERT on `events`
	FOR EACH ROW UPDATE courses SET modified=CURRENT_TIMESTAMP WHERE `id`=NEW.`course`;
CREATE TRIGGER `course_mod_upd` AFTER UPDATE on `events`
	FOR EACH ROW UPDATE courses SET modified=CURRENT_TIMESTAMP WHERE `id`=NEW.`course`;
CREATE TRIGGER `course_mod_del` AFTER DELETE on `events`
	FOR EACH ROW UPDATE courses SET modified=CURRENT_TIMESTAMP WHERE `id`=OLD.`course`;

CREATE TABLE `todo` (
    `id`      VARCHAR(36) PRIMARY KEY DEFAULT (uuid()),
    `user`    VARCHAR(36) NOT NULL,
    `name`    VARCHAR(64) NOT NULL,
    `completed` BOOL NOT NULL DEFAULT FALSE,

    FOREIGN KEY (`user`)
                  REFERENCES `users`(`id`)
                  ON UPDATE CASCADE ON DELETE CASCADE
);
